# SLSA Provenance Workflow

## Overview

Generates [SLSA Level 3](https://slsa.dev/spec/v1.0/levels) provenance
attestations for the release binary using the official
[SLSA GitHub Generator](https://github.com/slsa-framework/slsa-github-generator).
Triggered on every published GitHub Release.

**Workflow:** `.github/workflows/slsa-provenance.yml`  
**Trigger:** Release published, manual (`workflow_dispatch`)  
**Required secrets:** None (uses OIDC)  
**Permissions:** `contents: write`, `id-token: write`, `actions: read`

## What Is SLSA Provenance?

SLSA (Supply-chain Levels for Software Artifacts) provenance is a verifiable
record of **how** a software artifact was built: which source code, which
workflow, which builder, and under what conditions. SLSA Level 3 means the
provenance was generated by a trusted, isolated build platform (GitHub Actions)
and cannot be forged by the build script.

The provenance is stored as a signed attestation file alongside the release
assets.

## Jobs

### `build`

Builds the release binary and outputs its SHA-256 hash as a base64-encoded
string.

```bash
cargo build --release
sha256sum target/release/nsip | base64 -w0
```

The hash is passed to the `provenance` job as a workflow output.

### `provenance`

Calls the reusable workflow
`slsa-framework/slsa-github-generator/.github/workflows/generator_generic_slsa3.yml@v2.1.0`
(see the
[SLSA GitHub Generator releases](https://github.com/slsa-framework/slsa-github-generator/releases)
for available versions) with the base64-encoded subject hashes. The generator:

1. Verifies it is running in a trusted GitHub Actions environment
2. Signs the provenance with Sigstore keyless signing
3. Uploads the `.intoto.jsonl` provenance file to the release

## Verifying Provenance

```bash
# Install slsa-verifier
go install github.com/slsa-framework/slsa-verifier/v2/cli/slsa-verifier@latest

# Download the release binary and provenance
gh release download v0.6.0 --repo zircote/nsip --pattern '*'

# Verify (assets are bare, versioned binaries — no .tar.gz)
slsa-verifier verify-artifact \
  --provenance-path multiple.intoto.jsonl \
  --source-uri github.com/zircote/nsip \
  --source-tag v0.6.0 \
  nsip-0.6.0-linux-amd64
```

## Relationship to Other Attestation Mechanisms

| Mechanism | Workflow | Format | Level |
|-----------|---------|--------|-------|
| `actions/attest-build-provenance` | `release.yml` | Sigstore bundle | — |
| Cosign detached signatures | `signed-releases.yml` | `.sig` files | — |
| SLSA provenance | `slsa-provenance.yml` | `.intoto.jsonl` | SLSA Level 3 |

All three mechanisms are complementary and provide different verification paths.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| Provenance not attached | `upload-assets: true` flag missing | Verify the `with:` block in the `provenance` job |
| Verification fails | Binary was modified after provenance was generated | Do not modify release assets after provenance is generated |
| Generator version mismatch | Pinned to old `v2.1.0` | Check the [SLSA GitHub Generator releases](https://github.com/slsa-framework/slsa-github-generator/releases) for updates |
